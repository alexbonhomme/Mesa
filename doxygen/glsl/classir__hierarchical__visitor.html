Detailed Description

Base class of hierarchical visitors of IR instruction trees.

Hierarchical visitors differ from traditional visitors in a couple of important ways. Rather than having a single visit method for each subclass in the composite, there are three kinds of visit methods. Leaf-node classes have a traditional visit method. Internal-node classes have a visit_enter method, which is invoked just before processing child nodes, and a visit_leave method which is invoked just after processing child nodes.

In addition, each visit method and the accept methods in the composite have a return value which guides the navigation. Any of the visit methods can choose to continue visiting the tree as normal (by returning visit_continue), terminate visiting any further nodes immediately (by returning visit_stop), or stop visiting sibling nodes (by returning visit_continue_with_parent).

These two changes combine to allow nagivation of children to be implemented in the composite's accept method. The accept method for a leaf-node class will simply call the visit method, as usual, and pass its return value on. The accept method for internal-node classes will call the visit_enter method, call the accpet method of each child node, and, finally, call the visit_leave method. If any of these return a value other that visit_continue, the correct action must be taken.

The final benefit is that the hierarchical visitor base class need not be abstract. Default implementations of every visit, visit_enter, and visit_leave method can be provided. By default each of these methods simply returns visit_continue. This allows a significant reduction in derived class code.

For more information about hierarchical visitors, see:

http://c2.com/cgi/wiki?HierarchicalVisitorPattern http://c2.com/cgi/wiki?HierarchicalVisitorDiscussion

ir_visitor_status ir_hierarchical_visitor::visit ( class ir_dereference_variable *  ir)
virtual

ir_dereference_variable isn't technically a leaf, but it is treated as a leaf here for a couple reasons.

By not automatically visiting the one child ir_variable node from the ir_dereference_variable, ir_variable nodes can always be handled as variable declarations. Code that used non-hierarchical visitors had to set an "in a dereference" flag to determine how to handle an ir_variable. By forcing the visitor to handle the ir_variable within the ir_dereference_variable visitor, this kludge can be avoided.

In addition, I can envision no use for having separate enter and leave methods. Anything that could be done in the enter and leave methods that couldn't just be done in the visit method.
